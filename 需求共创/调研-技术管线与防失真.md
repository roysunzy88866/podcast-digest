**Whisper 长音频的已知坑**：静音/停顿会触发**无中生有和复读循环**（ACM FAccT 2024 记录；不同条件幻觉率 1%-80% 报告不一）。缓解：VAD 先切静音，或改用 gpt-4o-transcribe（开发者实测幻觉基本消失【个人实测口碑】）。来源：[Healthcare Brew 报道 FAccT 研究（2024-11）](https://www.healthcare-brew.com/stories/2024/11/18/openai-transcription-tool-whisper-hallucinations)、[Calm-Whisper 论文（2025-05）](https://arxiv.org/html/2505.12969v1)、[Zenn 开发者实测](https://zenn.dev/daishiro/articles/whisper-hallucination-gpt4o-transcribe?locale=en)

### A2. 说话人分离（diarization）

对访谈**很重要**：不分离说话人，「金句归属」会张冠李戴（把嘉宾的话安到主持人头上是典型失真）。
- 商用第一梯队：AssemblyAI、Deepgram、Speechmatics（[AssemblyAI 指南（2026）](https://www.assemblyai.com/blog/top-speaker-diarization-libraries-and-apis)）
- 访谈/播客黑马：Gladia Solaria-1，官方宣称 DER 低至竞品 1/3、分离含在基础价【官方宣称，未见独立复测】（[Coval 2026 对比](https://www.coval.ai/blog/best-speech-to-text-providers-in-2026-independent-benchmarks-and-how-to-choose/)）
- 开源：pyannote 3.1→4.0（2026），优化配置 DER ~10%；本地标准组合 WhisperX + pyannote（[MarkTechPost 指南（2025-08）](https://www.marktechpost.com/2025/08/21/what-is-speaker-diarization-a-2025-technical-guide-top-9-speaker-diarization-libraries-and-apis-in-2025/)）

### A3. YouTube 自动字幕 vs 自己转写

- 社区实测：YouTube 自动字幕干净英文 94-96%，但**无标点规范化、无说话人标签**，口音/术语/抢话掉到 78-85%；Whisper large-v3 约 95-97%（[NoteLM 实测（2026）](https://www.notelm.ai/blog/youtube-auto-captions-transcript)、[Aiville 对比](https://www.aiville.com/c/open-ai/audio-transcription-showdown-openai-s-whisper-vs-youtube-s-auto-subtitles)）
- 【推断】要做「金句保真+时间戳溯源」，YouTube 字幕没有说话人标签这条就出局——可当免费粗筛/索引层，正式管线自己转写。

## B. 浓缩/防失真环节（重点）

### B1. Quote-grounded / citation-based 忠实摘要的公开做法

1. **Anthropic Citations API**（2025-01 发布）：源文档按句切块，模型每个论断自动附「引用了哪几句原文」，**引用原文不计输出 token 费**；有用户报告来源幻觉 10%→0%【官方宣称+客户案例】。「每句摘要绑定原文引用」目前最现成的 API 级实现。（[官方博客](https://claude.com/blog/introducing-citations-api)、[Simon Willison 评述（2025-01-24）](https://simonwillison.net/2025/Jan/24/anthropics-new-citations-api/)）
2. **LongCite**（清华/智谱，[arXiv 2409.02897，2024-09](https://arxiv.org/abs/2409.02897)）：长上下文问答输出**句子级细粒度引用**，配套 LongBench-Cite 评测（[GitHub](https://github.com/THUDM/LongCite)）。
3. **FACTS Grounding 排行榜**（Google DeepMind，[arXiv 2501.03200，2025-01](https://arxiv.org/pdf/2501.03200)）：专测 LLM 长文落地忠实度，可用于选浓缩底模。
4. **「Correctness is not Faithfulness」**（[arXiv 2412.18004，2024-12](https://arxiv.org/pdf/2412.18004)）：引用「对」≠生成真的忠实——**引用要机器校验，不能只信模型自报**。
5. 工程实践：给 LLM 喂 **SRT/VTT 带时间戳格式**；「引语和时间戳必须回原转写稿核对」（[BrassTranscripts 指南（2025）](https://brasstranscripts.com/blog/powerful-llm-prompts-transcript-optimization)）

### B2. 长上下文一口吞 vs map-reduce 分段合并

- 2 小时访谈转写稿 ≈2.5-3 万 token，远小于现代模型窗口，**一口吞放得下**【推断，基于公开窗口规格】。
- 一口吞的风险在注意力不在窗口：长文摘要有**位置偏差**（文档中段忠实度显著更低，[NAACL 2025](https://aclanthology.org/2025.naacl-long.442.pdf)）；长输出**幻觉集中在末尾**（[arXiv 2505.15291，2025-05](https://arxiv.org/html/2505.15291)）。
- map-reduce 也不干净：[Context-Aware Hierarchical Merging（2025-02）](https://arxiv.org/pdf/2502.00977)直接结论——**层级合并即便最强 LLM 也做不出忠实摘要**，除非合并时把原文重新喂回；[BooookScore（ICLR 2024）](https://openreview.net/pdf?id=7Ttk3RzDeu)：层级合并更连贯但丢细节。
- 【推断】2 小时量级：「长上下文整读 + 强制引用时间戳 + 独立校验」优于 map-reduce。

### B3. 抽取式 + 生成式混合

- 学术侧：2024-2025 多篇「先抽关键句、再生成改写」混合框架（[PeerJ 统一框架（2025）](https://peerj.com/articles/cs-2424/)），共识：混合法在保留原文锚点上优于纯生成。
- 产品侧最有说服力：**Snipd**——金句以「原声片段+转写+AI 摘要」三件套存在，**原声片段零失真**，AI 只负责找位置写包装（[官网](https://www.snipd.com/all-features)、[TechCrunch（2022，机制沿用）](https://techcrunch.com/2022/08/16/how-snipd-is-using-ai-to-unlock-knowledge-in-podcasts/)）

### B4. 摘要忠实度自动评估

- 传统指标（SummaC 2021 / QAFactEval 2022）已被 LLM-as-judge 超越【2025 研究共识】。
- 2025 可用：**FaithBench + FaithJudge**（[Vectara 排行榜论文，EMNLP 2025](https://arxiv.org/pdf/2505.04847)）；**Q-S-E 框架**（[Nature Sci Reports（2025）](https://www.nature.com/articles/s41598-025-31075-1)）；警示：幻觉检测器进步被高估，LLM 裁判也会漏（[The Illusion of Progress（2025-08）](https://arxiv.org/html/2508.08285v2)）。
- 【推断】最实用的一层不用模型：**引语是否逐字出现在转写稿、时间戳区间是否真包含该句——字符串匹配就能机器卡死**，零成本零误报，放在 LLM 裁判之前。

### B5. NotebookLM / Podwise 可借鉴机制

- **NotebookLM**：RAG + 行内引用编号，悬停预览、点击跳源文档段落；无依据时答「无相关内容」。可抄：**每个论断可点击回跳原文**的交互契约。（[FSU 说明](https://servicecenter.fsu.edu/s/article/How-do-NotebookLM-s-inline-citations-work-and-why-are-they-important)、[EmergentMind（2025）](https://www.emergentmind.com/topics/notebooklm)）
- **Podwise**：摘要↔转写稿时间戳双向联动；用户头号抱怨是「**时间戳太粗、跳不准**」——时间戳精度=用户感知的可信度。（[Product Hunt 评价（2025-2026）](https://www.producthunt.com/products/podwise-ai/reviews)、[市场研究员评测](https://www.cascadeinsights.com/a-market-researchers-review-podwise)）

### C2. 「中文旁白 + 英文原声金句混排」现成产品

- **没找到全自动做这个形态的产品**【基于本次检索的推断，不排除遗漏】。最接近的拼图：Snipd（原声片段一等公民但无中文旁白）、ListenHub/NotebookLM（中文合成播客但不引原声）、播记（中文播客金句剪辑，面向创作者）。

假设：转写稿 2.5-3 万 token；LLM 跑 3-5 遍（大纲→抽金句→成稿→校验）≈10-15 万输入 + 2 万输出 token；中文旁白 1500-3000 字。

| 转写+分离（2h） | 本地 WhisperX ≈$0 / Groq turbo $0.08（无分离） | **AssemblyAI $0.34** | Deepgram ≈$1.16 |
| LLM 浓缩+校验 | DeepSeek ≈$0.02-0.05（[官方价](https://api-docs.deepseek.com/quick_start/pricing)） | **Gemini 2.5 Flash ≈$0.10**（[官方价](https://ai.google.dev/gemini-api/docs/pricing)） | 旗舰档 ≈$0.4-0.8 |

## 对本产品的启示

**推荐管线（人话版）**：
1. **转写**：AssemblyAI（一次拿到词级时间戳+说话人标签）；预算敏感 Groq turbo + 开源分离。**别用 YouTube 字幕做正式管线**（无说话人标签，金句会张冠李戴）。
2. **浓缩**：2 小时转写稿**一口吞整读，不要 map-reduce**（学术证据：层级合并即便最强模型也做不出忠实摘要）。输出强制「每个论断带 [时间戳+说话人] 引用」。
3. **TTS**：中文旁白首选 MiniMax speech-2.8 或豆包；金句**不翻译成中文念，直接按时间戳切英文原声插进去**——没人做、原声天然零失真，正好是「浓缩不失真」的杀手锏。

**每期成本**：主流配置 **$0.5-0.7（¥4-5）/期**，上限 $2.5。

**防失真最值得抄的 3 个做法**：
1. **Snipd 的「原声即金句」**：金句用原声片段承载而非转述——失真率物理上为零，比任何 AI 校验都硬。
2. **NotebookLM 的「句句可点回原文」**：中文精华稿每句绑时间戳，点击跳回转写稿原段+原音频位置（Podwise 用户头号抱怨就是跳不准——时间戳精度=感知可信度）。
3. **机器闸门在前、LLM 裁判在后**：第一级字符串匹配卡死「引语逐字存在于转写稿、时间戳区间真包含该句」（零成本不会漏）；第二级 FaithJudge/Q-S-E 式 LLM-as-judge 查改写段落有没有偷换意思。现成参考：Anthropic Citations API。
