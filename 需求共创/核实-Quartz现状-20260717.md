## 二、关键事实与证据

**① Quartz 版本现状** — 两个口径分开看:
- GitHub「Release 对象」口径:最新正式 release 仍是 **v4.0.8**(2023-08-21,标 latest)。别被它误导去装旧版。
- git tag 口径:**v5.0.0** 正式 tag 已于 2026-03-14 打出(commit `ab346fa`,「feat(v5): add plugin system」)。
- 默认分支 = `v5`,HEAD commit **2026-06-16**;文档站 whats-new 把 Quartz 5 当当前版本。
- 证据:`api.github.com/repos/jackyzha0/quartz/releases/latest`(v4.0.8)、`/tags`(v5.0.0)、`/branches/v5`(2026-06-16)。
- **对 C1**:按默认分支 clone 拿到的就是 v5;**为可复现须钉住确切 commit**。

**② Bases(实体页聚合地基)= 能用但绝不能盲押**:
- 「原生/开箱」**证伪**:官方原文「Bases support is provided by the BasesPage plugin」,须手动 `npx quartz plugin add github:quartz-community/bases-page`;whats-new 列在「Plugins new to v5」,不在默认配置。
- 承载插件 `quartz-community/bases-page`:**star 3、约 5 月龄(created 2026-02-22)、非核心作者(quartz-community 组织)维护**,有未关渲染 bug(卡片宽高比反渲染 #3、含空格视图名 transclusion 失效 #1)。
- 「自动聚合」**高估**:要为每个实体**手写 .base 文件**(query/视图/列)+ 每集先产出规范化 frontmatter;表达式引擎只支持 `if()/contains()/date()` 等函数子集。
- build 期 SSR 静态 HTML,与「静态站构建时聚合」用法匹配;但 `unlisted:true` 页被**排除在 base 视图外** → Cloudflare Access 场景须验实体聚合表不静默漏集。
- 证据:`quartz.jzhao.xyz/features/bases`、`github.com/quartz-community/bases-page`、`api.github.com/repos/quartz-community/bases-page/issues?state=open`。
- **对 C3(实体页)**:主区聚合**保留退路**(Dataview-Syncer 预烤 / 自建 build 期聚合脚本),不单押这个 3 星插件。

**③ 属性面板 vs frontmatter 聚合(解耦)**:
- Frontmatter 内置插件用 gray-matter 解析,`hidePropertiesView` 可隐藏 UI 但**仍解析** frontmatter。
- Bases 吃的是 frontmatter 数据本身,不依赖属性卡 UI 是否显示。
- **对项目**:真正需要的是 frontmatter 被 Bases 机器聚合,**不是**属性卡 UI。NoteProperties 现为外部组件 `quartz-community/note-properties`。

**④ Node 25.9 风险**:
- engines `node>=22`、`npm>=10.9.2`,**无上限** → 本机 Node 25.9 形式上满足,`npm i` 不被 engines 卡。
- 但上游 `.node-version=v22.16.0`(官方只实测 Node 22 LTS);Node 25 是奇数非 LTS,endoflife 记 EOL 2026-06-01(已过)。
- 尽力证伪「Node 25 build 不过」**失败**:sharp^0.34.5/@napi-rs/lightningcss 走 Node-API 按平台预编译(与 V8 ABI 解耦),esbuild^0.27.2 在 Node 25 无碍,未搜到 Quartz 因 Node 版本崩 build 的 issue → **无硬阻断,大概率能 build**。
- **建议**:优先切 Node 22.16.0(对齐上游)再真 build,把「未测试+EOL 运行时」变量剔掉;若用 Node 25.9 须把真 build 成/败结果落 docs 当证据。

## 三、正确初始化序列(官方 getting-started)

```
git clone (v5 默认分支) → cd quartz → npm i → npx quartz create
  → npx quartz plugin install --latest → npx quartz build --serve
```
- `npx quartz create` 是 clone 目录内的子步骤(bin→本地 `./quartz/bootstrap-cli.mjs`),**不是**脱离 clone 单跑的全局命令。
- **最可能让真 build 当场翻车、却与 Node 版本无关的坑**:fresh clone 的 `npx quartz plugin install` 因 `quartz.lock.json` 钉旧 commit(`@quartz-community/types` 等旧产物已不发布)会有插件 build 失败 → 官方解法 `--latest`(低内存机加 `-c 1` 一次装一个防 OOM)。

## 五、必须靠 C1 本机真 build 才能最终证实(联网核不死)

1. 所选 Node(25.9 或对齐后的 22.16.0)在本机 darwin arm64 上能否端到端真 build Quartz v5(成/败+日志)。
2. sharp/lightningcss/@napi-rs 是否命中 darwin-arm64 预编译,还是回落源码编译。
3. fresh clone 的 `npx quartz plugin install` 是否撞 lockfile 旧 commit,`--latest` 是否修好。
4. bases-page 能否真装 + 喂真 `.base` 后聚合表是否真渲染,未关 bug 是否命中所选视图。(C1 可暂不验,推 C3)
5. Cloudflare Access 下 unlisted/加密页是否让实体聚合表静默漏集。(推 C4/C7)
6. 钉哪个 commit 能得到可复现 build。
