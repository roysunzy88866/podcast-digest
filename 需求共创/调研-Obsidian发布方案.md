# 调研报告 · Obsidian 双链知识库 → 静态网站发布方案

> 调研日期：2026-07-16 · 独立调研 agent（官方文档 + GitHub 硬数据 + 社区实测）
> 触发：第 13 轮定位升级为「知识库」，用户点名 Obsidian 思路。
> 本文件是原始报告存档；架构决策见 需求共识.md。

## 0. 一句话结论

**Quartz v5 是唯一「知识层全家桶（双链+反链+图谱+中文搜索+标签）开箱即用、定制天花板够高」的免费方案**。但两个最特殊需求都**不该指望框架原生提供**：
- **实体页聚合金句** → 该由 AI 内容流水线生成 Markdown 时顺手写，不是网站框架的活
- **四层详情页+音频时间戳** → 在 Quartz 里写一个自定义组件即可（组件级，非重写级）

## 2. 关键判断题

### A. 定制天花板在哪
- **「实体页聚合金句」不是网站框架该干的活，是内容流水线该干的活**【推断，把握高】：任何 SSG 的聚合都停在「按标签查页面列表」；「把散在多集里关于某人的金句抽出来汇总」需要语义理解——**我们已有 AI 抽实体的流水线，让它生成每集 MD 时顺手把金句追加进对应实体页 MD**，实体页就退化成普通页面，反链/图谱自动生效。**把定制难度从「改框架」降为「改自己代码」。**
- **「四层 tab+音频时间戳跳转」没有现成方案原生提供，但 Quartz 内做得到**：v5 组件是 JSX 函数、可注入 JS/CSS，Page Type 插件定义一类页面怎么渲染。量级=一个组件+几十行 JS，做完搜索/反链/图谱/标签全保留。**代价：这是写 TS/JSX 不是改配置**（vibe coding 下由 AI 全权写、验收标准兜底）。

### B. 混合路线可行吗
可行有先例（[Rakshan Shetty 多站发布](https://rakshanshetty.in/blog/quartz-obsidian-multi-site-publishing) 证明同域族多静态站互跳在 CF 上成熟），**但作为 Plan B 而非首选**——既然详情页在 Quartz 内做得动，混合的隐性成本（搜索/图谱只覆盖一半、两套视觉对齐、slug 约定）不值得。它是「框架内做不动时」的逃生舱，逃生成本已验证不高。

### C. 推荐路线（人话）
**主体 Quartz v5，聪明活留在自己流水线：**
1. **流水线多生成一类 MD**：每集出 1 个集页 + 为每个实体维护 1 个实体页 MD（流水线往里追加出场记录+金句+[[双链]]）→ 实体页/反链/图谱/标签/时间排序全部白送
2. **Quartz 内定制「单集详情页」**：自定义组件做四层 tab+音频播放器+时间戳跳转（全项目唯一需写框架代码处，组件级）
3. **CF Pages 部署**（官方指南）；私密性 = 构建期 publish 过滤 + `_headers` noindex + **Cloudflare Access 挂登录门**（免费额度覆盖小圈子）。⚠️ **音频文件会随站点公开输出，门禁必须靠 CF Access，不能靠「没链接就找不到」**
4. **文件名即永久链接**：集页/实体页文件名用稳定 ID 生成、永不改名 → 绕开 Quartz 社区抱怨最多的「改名断链」
5. 日后详情页膨胀到 App 级（跨页持续播放/播放列表）→ 启用混合逃生舱

### 最大两个风险
1. **Quartz v5 太新**：正式 release tag 还停 v4（2023-08），v5 靠分支滚动，中文教程几乎全是 v4 的。深度定制后=维护带自定义组件的 fork，**上游大改的合并成本是长期税**。对冲：定制全收敛在独立插件/组件文件、不改核心。
2. **中文搜索质量是官方一面之词**：文档说 FlexSearch 正确分词中日韩【官方声明】，千页级全中文实测无第三方数据【未验证】。对冲：开工第一周灌 50 集真实数据做搜索验收，不行换 Pagefind 或自建索引（搜索在 Quartz 里是可替换插件）。

## 3. 对本项目的直接影响

- **验证了「知识库」定位技术上可落地，且大半白送** —— 实体页/反链/图谱/标签/搜索不用自己造
- **验证了架构分工**：知识层交给 Quartz，聪明活（实体抽取+金句聚合）交给已有 AI 流水线，唯一自建=单集详情页组件
- **私密性有解但有坑**：音频公开输出，必须 CF Access 加门（不是靠 URL 不可猜）
- **选型非最终**：Quartz v5 太新是真实风险，属【碰可移植】的重大判断，需用户拍板；且中文搜索需实测验收
