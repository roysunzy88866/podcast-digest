## A. 判官能力 vs 被判者能力

### A1. 「判官弱于被判者→评估质量下降」有硬证据

**[Benchmarks Saturate When The Model Gets Smarter Than The Judge](https://arxiv.org/abs/2601.19532)**（2026-01-27）【论文·摘要原文核对】：
> "Expert annotations reveal that Omni-Judge is wrong in **96.4% of the judge disagreements**... As problems become more challenging, we find that increasingly competent judges become essential in order to prevent judge errors from masking genuine differences between models."

- 小判官与强判官分歧时，**96.4% 是小判官错**
- benchmark 饱和是 **(数据集, 模型, 判官) 三元组**的属性，不是模型属性
- 模型排名会随判官更换而翻转【论文·正文，可靠度略低于摘要】

**佐证**：
- [S2J](https://arxiv.org/abs/2509.22099)（2025-09）：「solve-to-judge gap」——模型在**自己能解出**的题上仍有 **14%-37%** 判错；解不出的题判对概率更低 → **判断能力被解题能力封顶**
- [JudgeBench](https://arxiv.org/abs/2410.12784)（ICLR 2025）：需要客观正确性的困难 pair 上，**连 GPT-4o 都只比随机猜略好**（⚠️ 2024 论文，数字可能过时，但结构性结论仍被 2026 工作引用）

### A2. 弱判官能否发现强模型错误——「有条件地、部分地」

**[DeepMind: On scalable oversight with weak LLMs judging strong LLMs](https://arxiv.org/abs/2407.04622)**（2024-07）【论文】⚠️ 2024 资料，模型代际已变：
- debate（强 AI 对抗辩论、弱判官裁决）优于 consultancy
- **关键限制**：debate 只在「**信息不对称的抽取式 QA**」上明显占优，其他任务「results are mixed」

> **对本项目的意义**：我们的场景**恰好**是有信息不对称的抽取式验证（判官手里有原文，改写稿必须被原文支撑）——这是该文献里**最乐观的那一格**。这是原方案唯一真正的理论依据。

**[Representation-as-a-Judge（语义容量不对称假说）](https://arxiv.org/html/2601.22588)**（2026-07-09）【论文】：
- **评估所需语义容量 << 生成所需语义容量**
- 1.7B 小模型**探测内部表征**做二元分类 F1 **80-90%**；同一模型**用 prompt 直接问只有 10-25%**（多分类 50-60%）

> 🔑 **本次调研最关键的发现之一**：同一小模型「内部知道」和「说得出来」差 4-8 倍。
> - 好消息：小模型确实「懂」，评估任务本质比生成简单
> - **坏消息：我们只能用 prompt 这条路（API 不给 hidden states），而 prompt 恰是那个 10-25% 的路径**
> - 即「小模型能当判官」的乐观结论，**其实现方式恰好不是 API 用户能拿到的**

### A3. 小模型当 judge 的失败模式

**（1）Agreeableness bias —— 对本项目的致命一击**

[Beyond Consensus: Mitigating Agreeableness Bias](https://arxiv.org/abs/2510.11822)（2025-10）【论文·摘要原文】：
> LLM 评估者 **True Positive Rate 96%**，但 **"remarkably poor at identifying invalid ones (i.e., True Negative Rate <25%)"**

人话：**判官认出「好东西」96 分，认出「坏东西」不到 25 分。**

独立佐证（[eugeneyan, 2024-08](https://eugeneyan.com/writing/llm-evaluators/)）⚠️ 2024 资料【社区综述】：ChatGPT 识别 **>95% 的一致摘要**，但**只抓到 30-60% 的不一致摘要**。

> 🚨 **直接击中项目核心**：我们的主张是「不失真」，需要的是**抓出失真的能力（TNR/负类召回）**——而这恰恰是 LLM judge **最差**的维度。**判官在我们最需要它的地方最没用。**

**（2）判决不稳定**

[The Coin Flip Judge?](https://arxiv.org/abs/2606.13685)（2026-04-23）【论文】，测的正是小号模型（GPT-4o-mini / GPT-4.1-mini）：
- 成对偏好**平均 13.6% 翻转**；28% 的问题翻转率 >20%，最差一题 **56%**
- GPT-4o-mini **首位偏见**：72% A-majority（p=0.024）
- 两判官一致率仅 76%，Cohen's κ = **0.51**（仅「中等」）
- **语义等价的 prompt 模板改变 25% 的多数判决**
- 靠多数投票稳定复现需**平均 11 次重复**，高方差题 **15 次**

**（3）Self-preference bias（家族级）**
- [Self-Preference Bias](https://arxiv.org/abs/2410.21819)（2024-10）、[Quantifying and Mitigating](https://arxiv.org/abs/2604.22891)（2026-04）【论文】
- 社区实测【未同行评议，可靠度低】：平均自偏 +0.14；**家族层面成立**；**不需要「认出自己」**——模型识别自己作品只有随机水平却照样自偏

**（4）风格/流畅度敏感**【综述层面】
- 判官对 Markdown、列表、长度有偏好（verbosity/style bias）
- **对本项目尤其危险**：DeepSeek 写的中文稿一定结构漂亮读着顺——正好命中判官「看起来好就是好」的弱点，**而语义偷换恰恰藏在流畅行文里**

## B. 忠实度检测的 SOTA：专用小模型完胜通用大模型 judge

### B1. 数字（方向与 A 部分相反，但同样扎实）

**[LLM-AggreFact 排行榜](https://llm-aggrefact.github.io/)**（聚合 11 个 grounded factuality 数据集）【独立 benchmark】：

| 模型 | 规模 | 平均准确率 |
|---|---|---|
| **Bespoke-MiniCheck-7B** | 7B | **77.4%** |
| Claude-3.5 Sonnet | 巨大 | 77.2% |
| Mistral-Large 2 | 123B | 76.5% |
| gpt-4o-2024-05-13 | 巨大 | 75.9% |
| **Llama-3.1-405B** | **405B** | **70.4%** |

> **7B 打赢 405B。参数量在这个任务上几乎不预测性能。**

**[Vectara HHEM-2.1-Open](https://huggingface.co/vectara/hallucination_evaluation_model)**【官方宣称+可复现开源】：一个 **0.1B**（flan-t5-base 微调）模型，RAGTruth-Summ 64.42% vs GPT-4 62.62%、AggreFact-SOTA 76.55% vs GPT-4 73.78%——**打平或打赢 GPT-4**。

**[LettuceDetect](https://arxiv.org/abs/2502.17125)**（2025-02）【论文·结果表已核对】——**对原方案最直接的判决**：

| 方法 | RAGTruth example-level F1 |
|---|---|
| **LettuceDetect-large**（ModernBERT，~0.4B） | **79.22%** |
| 微调 Llama-2-13B | 78.7% |
| **Prompt GPT-4-turbo（零样本 LLM judge）** | **63.4%** |
| **Prompt GPT-3.5-turbo（零样本 LLM judge）** | **52.9%** |

> 🔑 **零样本 prompt 的 GPT-4-turbo 只有 63.4% F1，0.4B 专用模型有 79.22%。**
> 原方案正是「零样本 prompt 一个通用模型当判官」——**这条路即使用 GPT-4 也只有 63.4%；GLM-4.7-Flash 档次远在 GPT-4-turbo 之下，只会更低。**【推断，非实测】

### B2. 免费专用检测器的致命限制：**全是英文单语**

| 模型 | 体积 | 语言 |
|---|---|---|
| HHEM-2.1-Open | 0.1B（<600MB，CPU 2k token ~1.5 秒） | **仅英文** |
| Bespoke-MiniCheck-7B | 7B（MacBook 可跑，榜首 77.4%） | 英文为主 |
| LettuceDetect-large | ~0.4B，8k 上下文 | 英文 |
| Lynx-8B/70B | 8B/70B，HaluBench 87.3% | 英文 |
| **HHEM-2.3** | 商用**收费** | **11 语言含简中** |

> 🚨 **我们的任务是英文原文 → 中文精华稿 = 跨语言蕴含判定，上表免费的全部不能直接用。**唯一支持简中的 HHEM-2.3 是收费版。

**跨语言的额外风险**：
- [CCHall](https://aclanthology.org/2025.acl-long.1485.pdf)（ACL 2025）：**跨语言幻觉是独立的失败模式**，通过错误翻译产生（论文例：把 "stand" 译成「站在」而非「忍受」）——**这正是「偷换意思」在本项目里的典型长相**
- [Multilingual Self-Taught Faithfulness Evaluators](https://arxiv.org/abs/2507.20752)（2025-07）：合成多语训练的评估器**优于「先机翻再用英文检测器」**——说明机翻转接次优【是否开源未验证】
- [EMNLP 2023 多语摘要幻觉](https://aclanthology.org/2023.emnlp-main.551/)⚠️ 2023 资料：多语 LLM 在非英语中幻觉明显更频繁

### B3. 天花板警告

**LLM-AggreFact 榜首也只有 77.4%。这个任务没有「靠谱到可无人值守」的解——对所有方案一视同仁，包括推荐方案。**

## D. 工程实践建议（按证据强度排序）

### ✅ 强证据·优先做

1. **只做二元判定，不做打分**
   - [Decomposed Criteria-Based Evaluation](https://aclanthology.org/2025.emnlp-industry.136.pdf)（EMNLP 2025）：**二元判据 exact accuracy 87%**；序数（1-5 分）只有 **38-58%**
   - **判官只答「这句被原文支撑吗？是/否」，不问「质量如何打 1-5 分」**

2. **点对点喂原文片段，不做全局判断**
   - 与 DeepMind 结论一致：弱判官只在「信息不对称的抽取式」场景表现好
   - **不要把 15k token 原文+全稿一起丢过去问「有没有偷换意思」；要每个改写句配 800-1000 token 检索到的原文片段单独问**

3. **多次采样投票——但认清代价**
   - 需**平均 11 次重复**（高方差 15 次）才能 95% 稳定
   - **GLM 免费档 1 QPS**：一篇 30 段 × 11 次 = 330 次调用 = **至少 5.5 分钟纯排队**。「免费」变成时间成本。

4. **位置随机化 + 长度控制**（首位偏见 72% A-majority；prompt 措辞改变 25% 判决 → **judge prompt 措辞本身是未受控变量**）

### ⚠️ 证据混杂·不要想当然

5. **原子化分解不一定更好**
   - [Rethinking Atomic Decomposition for LLM Judges](https://arxiv.org/abs/2603.28005)（2026-03-30）：**holistic 判官在 4 个模型家族上都不劣于 atomic**，3 个家族有显著优势，且用 token 更少
   - **同篇最重要的发现**：「参考质量退化对两类判官都造成最大的准确率下降」→ **判官准确率主要取决于喂给它的原文片段检索得准不准，而不是判官多聪明**
   - **含义：与其纠结判官用 GLM 还是 DeepSeek，不如先把「给判官喂哪段原文」做对。这是投入产出比最高的一环。**

### 🔑 最有价值的一招【推断，但推导链条扎实】

6. **把弱判官当「单向否决票」，绝不当「通过证书」**
   - 依据：judge **TPR 96% / TNR <25%**
   - 逻辑：一个几乎从不 flag 的判官，**当它真的 flag 了，是高精确度信号**；但**它说「没问题」几乎不携带信息**
   - 同篇论文的 **minority-veto** 策略正是此思路
   - **GLM 说「有问题」→ 打回；GLM 说「没问题」→ 什么都不能推出，不记作通过**

### D2. 跨厂商独立性——有价值但证据有边界

- 自偏研究几乎全部是**开放式质量偏好任务**，不是「有原文可查的忠实度判定」
- **未找到任何研究直接测量「跨厂商独立性」与「判官能力」在忠实度任务上的权衡**【真实的证据空白】
- agent 判断【明确标注推断】：**能力差距（17 分）造成的漏检，很可能远大于同厂商自偏造成的漏检**——依据是 B1 的数量级差异（faithfulness 表现由「是否为该任务优化」主导，参数量和厂商都不是主要变量）

## 📌 对本项目的启示

### 一、最强的一环是机器闸门①，不是 LLM 裁判②

**「引语逐字存在于转写稿 + 时间戳区间包含该句」是整个方案里唯一不会说谎的东西**——没有 agreeableness bias、没有位置偏见、不会因 prompt 措辞改变 25% 判决。

> 按用户自己的纪律分级（能机器卡死的做成闸门 ＞ 卡不死的交第三方 ＞ 至少留锚点）：**①是真闸门，②是「第三方」，而现在选的第三方视力不太好。**

**第一优先级：把能塞进①的东西尽量塞进①。** 数字/日期/人名/机构名/专有名词 → **确定性抽取 + 逐一回原文比对**（数字必须在原文出现、人名必须在原文出现），零成本零幻觉，不交给 LLM。这类「机械可判」的失真最常见、最伤害「不失真」主张。

### 二、「DeepSeek 写稿 + 免费 GLM 裁判」**不足以支撑「不失真」主张**

| 问题 | 证据 |
|---|---|
| 能力落差落在论文警告区间 | GLM-4.7-Flash（AA 23）vs DeepSeek V4-Flash（AA 40），差 17 分 |
| **要的能力恰是判官最弱的** | 抓失真=要高 TNR；LLM judge **TNR <25%**（TPR 96%）|
| 选的路径天花板就低 | 零样本 prompt 通用模型：**GPT-4-turbo 仅 63.4% F1**；GLM 更低 |
| 跨语言是额外未测风险 | GLM-4.7-Flash 在**任何** faithfulness benchmark 上都无公开成绩 |
| 「偷换意思」正是最难那类 | CCHall：跨语言幻觉常表现为错误翻译；**语义偷换藏在流畅行文里，而判官偏爱流畅行文** |
| 免费的隐性成本 | 1 QPS，投票需 11-15 次 → 一篇排队 5 分钟起 |

> **一句话**：这个设计不是没有理论依据（我们的场景确实是 scalable oversight 最乐观的那一格）。**但能通过 API 拿到的实现路径（zero-shot prompt），恰好是语义容量不对称假说里表现 10-25% 的那条，而不是 80-90% 的那条。**
>
> 🚨 **它更像一个「让你感觉安全」的装置，而不是真的能拦住失真的闸门——这恰恰是最危险的一类设计。**

### 三、换判官几乎不要钱：**在用核心主张赌每篇一毛钱**

【推断/估算，基于官方定价】假设 30 个改写段落 × (1000 token 原文片段 + 200 token 段落) ≈ 36,000 输入 + 3,000 输出：

### 四、推荐架构

**方案 A（推荐）：加固① + 强判官② + 弱判官当否决票③**
1. **①层加厚**：引语逐字匹配 + 时间戳（已有）**＋ 数字/日期/人名/专名的确定性回原文比对**（新增，零成本零幻觉）
2. **②层判官换 DeepSeek V4-Pro**，且严格遵守：只做**二元判定** / **点对点喂原文片段** / **优先保证片段检索准确**（比判官选型影响更大）/ holistic 优先 / 位置随机化 + 长度控制
3. **GLM-4.7-Flash 降级为「跨厂商单向否决票」**：说有问题→打回；说没问题→**不记作通过**。利用跨家族独立性、规避 TNR 低的致命伤，且免费白捡。
4. 成本：每篇约 ¥0.13

**方案 B**：写稿 V4-Flash（AA 40），判官 V4-Pro（更强）——**判官 ≥ 写手**，这是 A1 论文的核心处方；同厂商自偏由 GLM 跨厂商否决票部分对冲。

**方案 C（不推荐但可行）**：坚持零成本 → 保留 GLM 判官，但**必须在产品主张上把「不失真」降级为「引语逐字可溯源 + 时间戳可核」**，不宣称改写段落经过可靠语义审查。**这是把主张对齐到证据，而不是把证据美化到主张。**

**明确不推荐**：
- ❌ HHEM-2.1-Open / MiniCheck / LettuceDetect —— **英文单语，我们跨语言用不了**（除非机翻回英文，而 [2507.20752](https://arxiv.org/abs/2507.20752) 说这条路次优）
