# ADR 0011 · C7b 全自动无人值守发布(用户知情确认,风险自担)

- **状态**:已定(2026-07-19 用户 AskUserQuestion 知情重拍)
- **相关**:drift #20 / #15 · story-map C7b · 与「里程碑亲手验收」纪律的冲突

## 背景
C7b(无人值守)= Workers 定时轮询 RSS → 自动跑完整条流水线 → 发布。发布形态三档:全自动一路发 / 半自动(成稿停「待发布」、人点头)/ 自动发+安全网。项目核心纪律是「里程碑用户亲手验收、不认机器绿」,而机器防失真闸门有**已知软肋**(D15 中文侧无硬拦、判官否决≈随机、D8/D17 覆盖天花板)。

## 决策
**全自动一路发布,不等人。** 用户 2026-07-19 在 AskUserQuestion 知情下**第二次**选定(drift #15 首次),选项已列明「坏稿可能直达公网、无人复核」。Claude 已交底风险并推荐半自动,用户仍选全自动。

## 风险与自担
- 若某集机器闸门未拦住坏稿(编造/失真),它会**直接发到完全公开的站点,发布前无人复核**。**用户明文接受此风险自担。**
- 这是「里程碑亲手验收」纪律的一个**明文例外,仅限 C7b 自动发布**;人工里程碑验收仍适用于其他切片 / 重大变更 / 标准改动。

## 工程兜底(不违背全自动,良性增强)
1. **闸门做到最严**:现有三联 + 事实 + 实体 + 关联 + 音频闸门,C7b 搬上 Workers 后逐条仍生效。
2. **发布后自动通知用户 + 保留一键撤下/回滚**(把选项 C 的安全网并入 B,不改「不等人」本质,只加事后补救)。
3. **定点重写回路**是无人值守下唯一的失真自愈手段(整篇重摇不收敛已证,docs/c3-定点重写回路.md)。
4. **失败告警**(闸门连续拦稿 / 流水线挂)通知用户(上线前 checklist 项,C7c 一并硬化)。

## 后续(C7b 架构可行性待核验,P1)
