# ADR 0008 · 实体页聚合改为自建(不押 Bases)

- **日期**:2026-07-17 · **状态**:**已采纳(决策不变)· 事实基础已修正,见文末「后续修正」** · **触发**:C3 开工前地基裁决(drift-log #1)

> ⚠️ **读之前先看文末〈后续修正 2026-07-17〉**:本 ADR 用来对比的「Bases=社区插件 vs 反链/图谱=真原生」这个二分**经实测证伪**,且「本就没装它」是**假的**(它已装、且默认启用)。**决策本身不受影响**(理由 2/3/4 与该二分无关),但事实段别照抄。

## 背景:被证伪的前提

需求共识 L159(🔒 第 21 轮据 Obsidian 调研写入)原文:

> · 用 **Bases**(Quartz v5 原生)做实体页聚合,不用 Dataview(Quartz 不原生)

**这句话的前提是假的。** C1 装机时对抗核实(`需求共创/核实-Quartz现状-20260717.md` §二②,GitHub API + 官方文档实查)证伪「原生」:

| 共识当初以为 | 实测 |
|---|---|
| Quartz v5 **原生内置** | **第三方社区插件** `quartz-community/bases-page`,须 `npx quartz plugin add` |
| (隐含)成熟可靠 | ⭐ **star 3**、月龄、**非 Quartz 核心作者**、**已知渲染 bug** |
| **自动**聚合 | 「自动」也要**每个实体手写一个 `.base` 配置**(线框规划 34 人物 + 21 公司 + 58 概念 ≈ 100+ 个) |

## 决策

**实体页聚合改为自建:流水线在生成期算好,直接写进实体页 markdown。**

即:每处理完一集,脚本自己算出「某人说过哪些金句、某实体出现在哪些集、常与谁同现」,把结果直接生成进实体页,**不依赖任何第三方插件做运行期聚合**。

## 为什么(依据,不是偏好)

1. **风险 vs 收益不成立**。押的是**产品命脉**——「知识串联/不让知识成孤岛」是本产品区别于市面所有竞品的核心(定位 v2 🔒 第 13 轮),而市面「浓缩」与「溯源」分裂、没人做「摘要级溯源+串联」(行业调研 §1)。把它押在一个 star 3、非核心作者、有已知 bug 的月龄插件上,插件坏/停维护/Quartz 升级不兼容 → 产品退化成普通翻译摘要站。
2. **「省代码」这个唯一好处本就不存在**。用 Bases 也要给 100+ 实体各写一个 `.base`,那必然得脚本生成 —— 既然都要写生成脚本,不如直接生成聚合结果,少一层第三方依赖。
3. **与既有架构认知一致**。需求共识 ③.7 自己就写着核心认知:「**知识关联不是网站魔法,是流水线的活**——自动层(反链/图谱)全靠流水线真写了显式链接+属性」。本决策只是把这条贯彻到底:聚合也归流水线。
4. **可移植**。生成的是纯 markdown,换发布层(Quartz→别的)不影响聚合结果。

## 代价(诚实记账)

- 我要多写聚合脚本(用户不用管)。
- 聚合是**生成期**算的静态结果,不是运行期动态查询 → 数据变了要重新生成。**影响可忽略**:本项目每次发布本就要重新 build。
- 放弃 Bases 可能提供的「运行期交互式筛选」(如页面上现场改筛选条件)。US-8/US-9 未要求此能力;US-2 的标签筛选由 Quartz 原生标签承担,不依赖 Bases。

## 后续修正(2026-07-17,C3 前置核验 P1 真 build 时实测发现)

**决策不变,事实基础错了两处。** 记在这里而不是抹掉原文——这个 ADR 本身就是为「纸面说原生、实测是插件」而写的,它自己再犯同类错误,更该留痕。

### 错误 1 ·「本就没装它」是假的

| ADR 原文说 | 实测(`site/`,2026-07-17) |
|---|---|
| 「Bases 插件**不装**」「**本就没装它**」 | **已装、且已启用**:`.quartz/plugins/bases-page` 存在、`quartz.lock.json` 有它、`quartz.config.yaml` L230-233 `enabled: true` |
| 「须 `npx quartz plugin add`」(=须手动 opt-in) | **不须**。`quartz.config.default.yaml` L213 里 Quartz **自带默认就 enabled**,而 `bootstrap-site.sh` L46 跑的是 `plugin install --from-config --latest` → **自动装上** |
| 「`bootstrap-site.sh` 无需变更」 | 反了:真要「不装」,**恰恰得改** config/bootstrap |

### 错误 2 ·「社区插件 vs 真原生」这个二分根本不成立

ADR 原文靠「Bases 是**第三方社区插件**,而反链/图谱/标签/搜索/双链是**真原生**」来立论。实测 `quartz.config.yaml`:

```
$ grep -o "source: github:[^/]*/" quartz.config.yaml | sort | uniq -c
  44 source: github:quartz-community/
```

**全部 44 个插件都是 `github:quartz-community/*`** —— 被 ADR 称作「真原生、白送」的那五个,来源与安装方式和 bases-page **完全同类**:

- `github:quartz-community/backlinks`(反链)· `github:quartz-community/graph`(图谱)
- `github:quartz-community/search`(搜索)· `github:quartz-community/tag-page`(标签)
- `github:quartz-community/obsidian-flavored-markdown`(双链渲染)

⇒ 在 Quartz v5 里,**几乎一切功能都是 `quartz-community` 插件经默认配置启用**。「原生 vs 社区插件」不是一条有意义的分界线;当初 需求共识 写「Bases=v5 原生」错了,本 ADR 反过来说它「是社区插件所以不可靠」**也没抓到要害**。

### 那么决策为什么依然成立

**真正的分界线不是「原生与否」,是这一个插件本身的成色与用法**——ADR 理由 1 里真正干活的是这几条(与「原生」二分无关):

- ⭐ **star 3、月龄、非核心作者、已知渲染 bug**(核实报告 §二②)——反链/图谱/搜索是成熟且被全站用户长期使用的,bases-page 不是。**这条区分是真的,只是不该叫「社区插件 vs 原生」,该叫「成熟 vs 生瓜」。**
- **理由 2**(「省代码」好处不存在:100+ 个 `.base` 也得脚本生成)、**理由 3**(与 ③.7「知识关联是流水线的活」一致)、**理由 4**(可移植)——**三条全部与该二分无关,独立成立**。

⇒ **自建聚合的决策不变**;drift-log #1 的裁决不变;C3 照 ADR 0008 做。

### 处置

- **bases-page 保持现状(装着、enabled),不专门去卸**。理由:①它是 Quartz 默认配置的一部分,卸它要改 config + 偏离上游默认 ②我们不建任何 `.base` 文件 → 它对本站**是惰性的**(本次真 build 已验:退出 0、无 error)③「不装」从来不是决策的必要条件,决策是「**不依赖它做聚合**」,这一条靠不写 `.base` 就已兑现 ④ 外科手术式改动:不顺手动不相关的东西。
- **顺带**:它随 D21(44 个插件 `--latest` 浮动)一起,C7 上云前钉版本时一并收敛。
- 落 drift-log #12。
